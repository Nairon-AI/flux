# Backward Trace Methodology

The systematic process for tracing from symptom to root cause. Adapted from [root-cause-tracing](https://skills.sh/bobmatnyc/claude-mpm-skills/root-cause-tracing).

> "NEVER fix where the error appears. ALWAYS trace back to find the original trigger."

---

## The Core Problem

Bugs rarely originate where they surface. An error in a rendering function might be caused by bad data from an API handler, which was caused by a missing validation in a parser, which was caused by an incorrect schema definition. If you fix the rendering function, you fix a symptom — the bug will reappear in a different form.

## The 5-Step Trace

### Step 1: Observe the Symptom

Document exactly what failed:
- The exact error message or incorrect behavior
- The file and line where the error surfaced
- The operation that triggered it
- The state of the system when it happened

```
SYMPTOM:
  Error: "Cannot read property 'id' of undefined"
  Location: src/components/UserCard.tsx:42
  Trigger: Clicking "View Profile" button
  State: User logged in, viewing team page
```

### Step 2: Find the Immediate Cause

Identify the code directly producing the error. Read the file, find the exact line.

```
IMMEDIATE CAUSE:
  Line 42: const userId = user.id
  Problem: `user` is undefined at this point
  Question: Where does `user` come from?
```

### Step 3: Trace One Level Up

Follow the data backward. Where did the problematic value come from?

- If it's a function parameter → who called this function?
- If it's from state/store → who set this state?
- If it's from an API response → what did the API return and why?
- If it's from a database → what query produced this result?

```
LEVEL UP:
  `user` comes from: props.user (passed by parent component)
  Parent: src/pages/TeamPage.tsx:78
  Code: <UserCard user={teamMembers[index]} />
  Problem: teamMembers[index] is undefined — index is out of bounds
  Question: Why is index out of bounds?
```

### Step 4: Continue Backward

Keep going. At each level, ask: "Is this the origin, or is this just relaying bad data?"

**Signs you've found the root cause:**
- This is where a wrong decision was made (wrong logic, wrong assumption)
- This is where external data entered the system without validation
- This is where a contract was violated (function expected X, got Y)
- Fixing here would prevent the bug AND all downstream symptoms

**Signs you need to keep tracing:**
- This code is just passing data through — it didn't create the problem
- The data was already wrong when it arrived here
- Adding a fix here would be a band-aid, not a cure

```
LEVEL UP:
  `index` comes from: URL parameter parsed in useParams()
  Code: const { index } = useParams()
  Problem: URL has index=5 but team only has 3 members
  Question: Who generates these URLs?

LEVEL UP:
  URLs generated by: src/utils/teamLinks.ts:15
  Code: return `/team/${team.id}/member/${memberIndex}`
  Problem: memberIndex is calculated from stale data — the team roster
  was cached, a member was removed, but the cache wasn't invalidated

  ROOT CAUSE FOUND: Cache invalidation missing when team members change
  Location: src/services/teamCache.ts — no invalidation on member removal
```

### Step 5: Fix at Source + Add Defense

**Primary fix**: Fix the root cause.
```
Fix: Add cache invalidation in teamCache.ts when members are added/removed
```

**Defense-in-depth** (optional, at trust boundaries):
```
Defense 1: TeamPage.tsx — validate index is within bounds before rendering
Defense 2: useParams() wrapper — validate numeric params are within expected range
```

Defense-in-depth is not a substitute for the root cause fix. It's an additional safety net at layers where data crosses boundaries.

## Trace Documentation Template

Use this template during investigation. Each level becomes evidence in the RCA report.

```
## Trace: [Bug title]

### Symptom
- Error: [exact message]
- Location: [file:line]
- Trigger: [user action or system event]

### Level 0: Error Site
- File: [file:line]
- Code: [relevant snippet]
- Problem: [what's wrong at this level]
- Data source: [where the bad data comes from]

### Level 1: [Caller / Data Source]
- File: [file:line]
- Code: [relevant snippet]
- Problem: [what's wrong at this level]
- Data source: [where it comes from at this level]

### Level N: ROOT CAUSE
- File: [file:line]
- Code: [relevant snippet]
- Root cause: [why this is the origin]
- Fix: [what needs to change]

### Chain Summary
[Root cause] → [Level N-1] → ... → [Level 0: Symptom]
```

## Common Root Cause Patterns

Recognizing patterns speeds up tracing:

| Pattern | Symptom | Root cause is usually... |
|---------|---------|--------------------------|
| Undefined/null access | TypeError deep in rendering | Missing data validation at API boundary or store update |
| Stale data | UI shows old information | Missing cache invalidation or subscription |
| Race condition | Intermittent failures | Missing synchronization or wrong assumption about ordering |
| Type mismatch | Wrong output, no error | Implicit coercion at a boundary (string vs number, etc.) |
| Off-by-one | Wrong item selected/missing | Index calculation or boundary check |
| Missing error handling | Unhandled promise rejection | Async call without catch/try at the call site |
| Config drift | "Works on my machine" | Environment-specific value not in config management |

---

## Red Flags — Stop and Keep Tracing

If you catch yourself doing any of these, you haven't found the root cause yet:

1. **"Let me just add a null check here"** — you're patching a symptom
2. **"This quick fix should prevent it"** — you don't know WHY it happened
3. **"The trace is getting complicated, let me just..."** — complexity means the bug is real and deep
4. **"I think I know what's wrong"** — thinking is not tracing. Follow the evidence.
5. **"It probably won't happen again"** — if you can't explain why, it will

## When the Trace Gets Stuck

Sometimes you can't easily follow the chain:

- **Async boundaries**: Add logging/instrumentation to capture the data at each stage
- **Third-party code**: Read the library source or docs to understand what it does with your input
- **Dynamic dispatch**: Trace which concrete implementation runs at the call site
- **Event-driven systems**: Find all publishers and subscribers for the event
- **Database**: Check the query, the data, and what wrote the data

If truly stuck, bisect: find the last known-good state and the first known-bad state, narrow down what changed between them.
